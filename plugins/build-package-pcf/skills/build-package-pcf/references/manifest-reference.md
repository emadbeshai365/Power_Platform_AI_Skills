## Contract-first workflow

1. Identify target hosts and component architecture.
2. Define field/dataset bindings and maker inputs.
3. Define outputs/events and their commit semantics.
4. Declare resources and platform features.
5. Add localization keys.
6. Build to regenerate `IInputs`/`IOutputs`.
7. Implement against generated types.

Never edit `generated/ManifestTypes.d.ts`. Treat a build error after a manifest change as a contract error first, not a reason to weaken types.

## Control identity

The control identity combines namespace and constructor. Keep both stable after deployment.

```xml
<control
  namespace="Contoso.Controls"
  constructor="StatusPicker"
  version="1.0.0"
  display-name-key="StatusPicker_DisplayName"
  description-key="StatusPicker_Description"
  control-type="virtual">
```

- Use `standard` for `StandardControl`.
- Use `virtual` only for a project created with `--framework react` and implemented as `ReactControl`.
- Advance version intentionally for distributable updates.
- Use RESX keys rather than hard-coded manifest labels.

## Properties and outputs

```xml
<property
  name="value"
  display-name-key="Value_DisplayName"
  description-key="Value_Description"
  of-type="SingleLine.Text"
  usage="bound"
  required="true" />
```

### Usage

| Usage | Meaning |
| --- | --- |
| `bound` | Host-bound value that can participate in inputs and outputs. |
| `input` | Maker configuration or read-only input; do not return it from `getOutputs`. |
| `output` | Output-only contract where supported by the schema/host. |

Do not infer output behavior from the TypeScript type alone. Confirm the manifest usage and intended host.

### Object outputs

Object outputs require an explicit runtime schema:

```xml
<property
  name="Data"
  display-name-key="Data_DisplayName"
  description-key="Data_Description"
  of-type="Object"
  usage="output"
  required="true" />
```

- Implement `getOutputSchema(context)` and return a schema for every Object output.
- Use only Microsoft's supported JSON Schema subset.
- For Canvas, follow the current Object Output sample: define the hidden schema input and connect it to the output with `property-dependencies`.
- Treat the object shape as a versioned public contract. Avoid unbounded or framework-owned objects.

### Common type families

- Text: `SingleLine.Text`, `SingleLine.Email`, `SingleLine.Phone`, `SingleLine.URL`, `Multiple`.
- Numeric: `Whole.None`, `Decimal`, `FP`, `Currency`.
- Boolean/date: `TwoOptions`, `DateAndTime.DateOnly`, `DateAndTime.DateAndTime`.
- Choices: `OptionSet`, `MultiSelectOptionSet`.
- Lookup: `Lookup.Simple`.

Check the current type reference before selecting a type. Do not copy an `of-type` from a sample without confirming host and column compatibility.

### Type groups

Use a type group only when one implementation intentionally supports every listed type:

```xml
<type-group name="NumericTypes">
  <type>Whole.None</type>
  <type>Decimal</type>
  <type>FP</type>
  <type>Currency</type>
</type-group>

<property
  name="value"
  display-name-key="Value_DisplayName"
  of-type-group="NumericTypes"
  usage="bound"
  required="true" />
```

At runtime, use parameter metadata/formatting rather than assuming identical display behavior.

## Datasets and property sets

```xml
<data-set
  name="dataSet"
  display-name-key="DataSet_DisplayName"
  cds-data-set-options="displayCommandBar:true;displayViewSelector:true">
  <property-set
    name="titleColumn"
    display-name-key="TitleColumn_DisplayName"
    of-type="SingleLine.Text"
    usage="input"
    required="true" />
</data-set>
```

Use property sets to express semantic column roles such as title, start date, category, image, status, amount, or indicator. This makes a dataset component reusable across tables and views.

Review:

- required versus optional roles;
- compatible column types;
- missing/secured/null column behavior;
- command bar and view-selector ownership;
- host availability for dataset/property-set elements.

## Resources and React platform libraries

Standard example:

```xml
<resources>
  <code path="index.ts" order="1" />
  <css path="css/StatusPicker.css" order="1" />
  <resx path="strings/StatusPicker.1033.resx" version="1.0.0" />
</resources>
```

React virtual example generated by current PAC templates includes platform libraries:

```xml
<resources>
  <code path="index.ts" order="1" />
  <platform-library name="React" version="16.14.0" />
  <platform-library name="Fluent" version="9.46.2" />
</resources>
```

Rules:

- Keep resource paths relative to the manifest control directory.
- Declare every runtime CSS, RESX, and supported image resource.
- For React virtual controls, preserve versions generated by a current PAC CLI unless current Microsoft documentation requires a change.
- Do not specify Fluent 8 and Fluent 9 simultaneously.
- Remove the Fluent platform-library entry if the component does not use Fluent.
- Do not bundle a second React runtime into a virtual control.

## Feature usage and external services

Declare model-driven features used by the component:

```xml
<feature-usage>
  <uses-feature name="WebAPI" required="true" />
  <uses-feature name="Device.pickFile" required="false" />
</feature-usage>
```

Set `required="true"` only when the component cannot provide a meaningful degraded experience.

For directly contacted external services where the schema supports it:

```xml
<external-service-usage enabled="true">
  <domain>api.contoso.com</domain>
</external-service-usage>
```

Declare every contacted domain. Explain premium licensing and client-side security implications. Do not put secrets or bearer tokens in manifest properties.

## Events and host-specific elements

Custom events are currently pre-release for model-driven and Canvas apps:

```xml
<event
  name="onCommit"
  display-name-key="OnCommit_DisplayName"
  description-key="OnCommit_Description" />
```

Build after adding events, raise the generated `context.events.onCommit()` callback intentionally, and document the Power Fx or model-driven handler configuration. Do not promise Power Pages support.

Canvas-only manifest behavior includes `property-dependencies`, `property-dependency`, and `pfx-default-value`. Availability changes independently by element; always check the **Available for** section of the current schema page.

Do not add a manifest node merely because it exists. Tie it to an implemented behavior and a supported host.

## Dependent libraries and platform actions

Dependent libraries are preview. Record preview acceptance and isolate the behavior before using them.

```xml
<platform-action action-type="afterPageLoad" />
```

`platform-action` is model-driven only. Do not include it in a Canvas or Power Pages contract.

## Manifest review checklist

- Namespace/constructor stable and syntactically valid.
- Version intentionally advanced.
- `control-type` matches implementation interface.
- Every property has correct usage, type, requirement, and localization keys.
- `getOutputs` returns only declared outputs.
- Every Object output has a valid `getOutputSchema` entry and required Canvas dependency.
- Every event has localized metadata, generated typing, documented handler wiring, and preview acceptance.
- Dataset roles are semantic and reusable.
- All resource files exist and are declared once in deterministic order.
- React/Fluent platform libraries match current supported ranges.
- Feature usage matches actual API calls.
- External domains are complete and licensing impact is documented.
- Every element is supported by every promised host or guarded by a host-specific variant.
- Canvas-only defaults/dependencies and model-driven-only platform actions do not leak into incompatible variants.
- Build regenerates types without errors.
